# AGENTS.md

## 沟通要求

- 输出尽量分块，方便快速扫读和判断。
- 解释管理、系统、流程等偏抽象内容时，可少量使用有逻辑提示作用的 emoji 作为标题，例如：📌 结论、🧩 拆解、⚠️ 风险、🔧 要补、📊 进度、👀 待拍板。
- 列点时使用轻量符号辅助扫读，例如：✅ 已有、➡️ 下一步、- 普通列点、1. 顺序事项。
- emoji 和符号只服务于结构和理解，不做装饰堆砌。

## 工作要求

- 每完成一个独立任务后，都要做一轮 commit。
- commit 应只包含本次任务相关改动，避免把无关内容混在一起。
- commit message 要简洁明确，能直接说明这次完成了什么。

## codex-tasks 留言跟踪

- `codex-tasks/` 是 Claude / DeepSeek 给 Codex 留任务的地方，由夏董负责转告，不需要 agent 之间直接沟通。
- Codex 处理某个任务文件后，在原文件里直接勾选已完成事项，并补一段 `Codex 处理记录`。
- 处理记录只需要写清楚：改了什么、怎么验证、对应 commit。
- 如果任务没法完成，就在原文件里写 `卡住原因` 和 `需要夏董拍板`，不要另起复杂流程。
## 代码结构规范

### 防御性编码禁令
- 不在上下游重复校验同一条件。只在 system boundary（用户输入、外部 API）做验证，不信任调用方时用 assert 而非 if-guard。
- 写函数前先 grep 所有调用方，确认参数的实际取值范围，不为不存在的场景做 fallback。
- 函数签名不要为了“以后可能有别的调用方”而放宽类型（如 `str | None`），只反映当前实际调用链。
- 重构时新代码上线后，旧代码必须删除。不许留尸体。

### Schema / 常量定义

- 同一份信息（表名、字段列表、列类型）只在一个地方定义，其他模块引用。不接受维护两套几乎一样的 dict。

### 代码审查自检

每次写代码后，自问：
- 这个检查是不是上游已经做过了？
- 这个 null guard 是不是真会触发？
- 这个 fallback 路径是不是真有调用方会走？
- 有没有刚被我替换掉但忘记删的旧函数？

如果任一答案是“是”，先改掉再提交。
